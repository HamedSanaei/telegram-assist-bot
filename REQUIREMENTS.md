# سند نیازمندی‌های ربات دستیار مدیریت کانال‌های تلگرام

## 1. معرفی پروژه

هدف پروژه، توسعه یک ربات دستیار ادمین با زبان Python است که بتواند محتوای کانال‌های عمومی تلگرام را جمع‌آوری، پردازش، دسته‌بندی و برای بررسی در اختیار مدیران قرار دهد.

این برنامه در ابتدا برای مدیریت محتوای چند کانال تلگرام استفاده خواهد شد، اما معماری آن باید به‌گونه‌ای طراحی شود که در آینده بتوان قابلیت‌های جدیدی مانند تحلیل عملکرد پست‌ها، بازنویسی محتوا، مدیریت تبلیغات، انتشار خودکار، داشبورد مدیریتی و پشتیبانی از چند حساب تلگرام را بدون بازنویسی بخش‌های اصلی به آن اضافه کرد.

دیتابیس اصلی پروژه MongoDB است.

پروژه باید به‌صورت ماژولار، قابل تست و قابل توسعه پیاده‌سازی شود و وابستگی مستقیم منطق اصلی برنامه به تلگرام، MongoDB یا سرویس‌های هوش مصنوعی به حداقل برسد.

## 2. اهداف اصلی برنامه

برنامه باید قابلیت‌های زیر را فراهم کند:

- ورود به تلگرام با یک حساب کاربری دارای Telegram Premium
- ذخیره و استفاده مجدد از Session حساب تلگرام
- پایش کانال‌های عمومی مشخص‌شده در فایل تنظیمات
- دریافت پیام‌های منتشرشده در روز جاری
- دریافت لحظه‌ای پیام‌های جدید کانال‌های مبدا
- ذخیره متن، مدیا، متادیتا و Entityهای هر پیام
- جلوگیری از ذخیره و پردازش دوباره یک پیام
- نگهداری پست‌ها به مدت ۱۴ روز
- تشخیص محتوای تبلیغاتی با استفاده از سرویس هوش مصنوعی
- تشخیص محتوای تکراری یا مشابه
- حذف نام کاربری‌ها و لینک‌های نامرتبط تلگرام
- ارسال پست‌های پیشنهادی برای مدیران
- امکان انتشار فوری یا زمان‌بندی‌شده در کانال‌های مقصد
- حفظ Premium Emoji و Custom Emojiهای تلگرام
- هماهنگ بودن دکمه‌ها و وضعیت پست‌ها میان تمام مدیران
- ثبت کامل وضعیت پردازش و انتشار هر پست

## 4. مدیریت تنظیمات

تنظیمات برنامه در فایل `configuration.json` نگهداری می‌شوند.

این فایل باید حداقل شامل بخش‌های زیر باشد:

- اطلاعات اتصال MongoDB
- مسیر یا نام فایل Session تلگرام
- اطلاعات موردنیاز برای ورود حساب تلگرام
- توکن Telegram Bot
- فهرست مدیران مجاز
- فهرست کانال‌های مبدا
- فهرست کانال‌های مقصد
- کانال یا گروه تایید محتوا
- تنظیمات سرویس‌های هوش مصنوعی
- فعال یا غیرفعال بودن تشخیص تبلیغات
- فعال یا غیرفعال بودن تشخیص محتوای تکراری
- فعال یا غیرفعال بودن امتیازدهی هوش مصنوعی
- فاصله زمانی انتشار پست‌های زمان‌بندی‌شده
- منطقه زمانی برنامه
- تنظیمات Logging
- تنظیمات پست‌های تبلیغاتی

اطلاعات حساس مانند موارد زیر نباید مستقیماً در فایل تنظیمات موجود در GitHub قرار بگیرند:

- API Key
- Telegram API Hash
- Telegram API ID
- Bot Token
- رمز MongoDB
- توکن سرویس‌های هوش مصنوعی

این اطلاعات باید از Environment Variable، Secret Manager یا فایل تنظیمات محلی خارج از Git خوانده شوند.

در زمان شروع برنامه، فایل تنظیمات باید اعتبارسنجی شود. اگر تنظیمات ضروری ناقص یا نامعتبر باشند، برنامه باید با پیام خطای واضح متوقف شود.

## 5. فاز اول: خزش، پردازش و انتشار محتوای کانال‌ها

### 5.1 ورود به حساب تلگرام

در اولین اجرای برنامه، حساب کاربری دارای Telegram Premium باید با شماره تلفن وارد تلگرام شود.

پس از ورود موفق:

- Session حساب باید ذخیره شود.
- در اجرای مجدد برنامه نباید دوباره کد ورود درخواست شود.
- Session باید به‌صورت امن نگهداری شود.
- فایل Session نباید در GitHub قرار بگیرد.
- در صورت منقضی یا باطل شدن Session، برنامه باید خطای قابل‌فهم ثبت کند.
- قطع موقت اینترنت نباید باعث حذف یا خراب شدن Session شود.

برای خزش و انتشار محتوا باید از Telegram User API و همان حساب دارای Premium استفاده شود.

Telegram Bot API فقط برای ارتباط با مدیران، دریافت فرمان‌ها و نمایش دکمه‌های تایید استفاده خواهد شد.

### 5.2 کانال‌های مبدا

فهرست کانال‌های مبدا در فایل `configuration.json` تعریف می‌شود.

برای هر کانال مبدا، اطلاعات زیر باید قابل تنظیم باشد:

- شناسه یا Username کانال
- فعال یا غیرفعال بودن کانال
- دسته‌بندی پیش‌فرض کانال
- فعال بودن بررسی تبلیغاتی
- فعال بودن بررسی محتوای تکراری
- حداقل امتیاز لازم برای پیشنهاد محتوا
- کانال‌های مقصد مجاز برای محتوای این کانال

پس از اجرای برنامه، سیستم باید برای هر کانال فعال:

1. پیام‌های منتشرشده در روز جاری را دریافت کند.
1. پیام‌های جدیدی را که پس از شروع برنامه منتشر می‌شوند به‌صورت لحظه‌ای دریافت کند.
1. قبل از ذخیره، بررسی کند که پیام قبلاً پردازش نشده باشد.
1. اطلاعات پیام را در MongoDB ذخیره کند.

منظور از «روز جاری» باید بر اساس منطقه زمانی تعیین‌شده در تنظیمات باشد. منطقه زمانی پیش‌فرض می‌تواند Asia/Tehran باشد.

مرز شروع روز باید دقیقاً ساعت 00:00:00 همان منطقه زمانی در نظر گرفته شود.

### 5.3 جلوگیری از پردازش تکراری

هر پیام باید بر اساس ترکیبی از اطلاعات زیر به‌صورت یکتا شناسایی شود:

- شناسه عددی کانال مبدا
- شناسه پیام در کانال مبدا

در MongoDB باید Unique Index برای این دو مقدار ایجاد شود.

حتی در شرایط زیر نیز نباید پیام دوباره پردازش شود:

- Restart شدن برنامه
- قطع و وصل شدن اینترنت
- دریافت هم‌زمان پیام از خزش اولیه و Listener لحظه‌ای
- اجرای چند Worker
- تکرار رویدادهای تلگرام
- خطا و Retry شدن عملیات

عملیات پردازش پیام باید Idempotent باشد.

### 5.4 ذخیره اطلاعات پست

برای هر پست حداقل اطلاعات زیر باید ذخیره شود:

- شناسه داخلی پست
- شناسه عددی کانال مبدا
- Username کانال مبدا
- نام نمایشی کانال مبدا
- شناسه پیام مبدا
- متن اصلی پیام
- Caption اصلی
- متن پردازش‌شده
- Entityهای تلگرام
- Premium Emoji و Custom Emoji Entityها
- نوع مدیا
- اطلاعات فایل یا مدیا
- شناسه Media Group
- زمان انتشار در کانال مبدا
- زمان دریافت توسط سیستم
- دسته‌بندی محتوا
- وضعیت تبلیغاتی بودن
- نتیجه بررسی محتوای تکراری
- امتیاز هوش مصنوعی
- وضعیت پردازش
- وضعیت ارسال برای تایید
- وضعیت انتشار
- کانال‌های مقصد انتخاب‌شده
- اطلاعات زمان‌بندی
- آخرین خطا
- تعداد تلاش‌های پردازش
- زمان ایجاد و آخرین بروزرسانی رکورد

پست‌ها باید حداکثر ۱۴ روز در دیتابیس نگهداری شوند.

برای حذف خودکار پست‌های منقضی‌شده باید در MongoDB یک TTL Index ایجاد شود.

مبنای TTL باید فیلدی مانند expires_at باشد که هنگام ذخیره هر پست برابر با ۱۴ روز پس از زمان دریافت پست تنظیم می‌شود.

### 5.5 مدیریت مدیا

برنامه باید حداقل از انواع مدیای زیر پشتیبانی کند:

- عکس
- ویدیو
- فایل
- صوت
- Voice
- Animation یا GIF
- Sticker
- Video Note
- آلبوم یا Media Group
- پست متنی بدون مدیا

اطلاعات مدیا باید به‌گونه‌ای ذخیره شود که پست تا ۱۴ روز بعد قابل ارسال مجدد باشد.

اگر استفاده مجدد از شناسه فایل تلگرام امکان‌پذیر نباشد، فایل باید در فضای ذخیره‌سازی محلی یا یک Object Storage دانلود و نگهداری شود.

برای هر فایل باید موارد زیر ثبت شوند:

- نوع فایل
- نام فایل
- MIME Type
- حجم فایل
- شناسه فایل تلگرام
- مسیر ذخیره‌سازی
- Hash فایل
- وضعیت دانلود
- تاریخ انقضا

در صورت حذف رکورد پست پس از ۱۴ روز، فایل‌های محلی وابسته به آن نیز باید پاک شوند.

### 5.6 مدیریت آلبوم‌ها

پیام‌هایی که بخشی از یک Media Group هستند باید به‌عنوان یک پست واحد پردازش شوند.

برنامه نباید هر عکس یا ویدیوی یک آلبوم را به‌صورت جداگانه برای مدیر ارسال کند.

برای دریافت کامل آلبوم، سیستم می‌تواند مدت کوتاهی منتظر بماند تا تمام اعضای Media Group دریافت شوند و سپس آن‌ها را به ترتیب صحیح ذخیره و ارسال کند.

### 5.7 حفظ Premium Emoji

Premium Emoji و Custom Emojiهای موجود در متن باید حفظ شوند.

برای حفظ آن‌ها:

- متن و Telegram Entityها باید به‌صورت کامل ذخیره شوند.
- موقعیت Entityها پس از اصلاح متن باید مجدداً محاسبه شود.
- انتشار در کانال‌های مقصد باید با حساب کاربری دارای Telegram Premium انجام شود.
- برای انتشار نهایی نباید از Telegram Bot API استفاده شود.
- ترتیب، Offset و Length مربوط به Entityها نباید خراب شود.

پس از حذف Username یا لینک از متن، Offset تمام Entityهای باقی‌مانده باید به‌درستی بروزرسانی شود.

### 5.8 بررسی تبلیغاتی بودن پست

در صورت فعال بودن این قابلیت در فایل تنظیمات، متن پست باید برای تشخیص تبلیغاتی بودن به سرویس هوش مصنوعی ارسال شود.

نتیجه باید حداقل شامل موارد زیر باشد:

- تبلیغاتی است یا خیر
- میزان اطمینان
- دلیل کوتاه
- نام مدل استفاده‌شده
- زمان بررسی
- نسخه Prompt

در صورت تشخیص تبلیغاتی بودن پست:

- پست نباید وارد ادامه Pipeline عادی شود.
- وضعیت آن در دیتابیس باید RejectedAsAdvertisement ثبت شود.
- در صورت فعال بودن تنظیم مربوطه، می‌توان خلاصه‌ای از دلیل رد شدن را برای مدیران ارسال کرد.

در صورت قطع بودن سرویس هوش مصنوعی، رفتار سیستم باید قابل تنظیم باشد:

- ادامه پردازش پست
- توقف پردازش پست
- قرار دادن پست در صف بررسی مجدد
- ارسال پست برای بررسی دستی

تمام درخواست‌های سرویس هوش مصنوعی باید Timeout و Retry محدود داشته باشند.

### 5.9 بررسی محتوای تکراری

سیستم باید پست جدید را با پست‌های ۱۴ روز گذشته مقایسه کند.

بررسی تکراری بودن باید در دو سطح انجام شود:

#### تکرار دقیق

برای تشخیص متن کاملاً یکسان یا بسیار نزدیک می‌توان از موارد زیر استفاده کرد:

- متن Normalizeشده
- Hash محتوا
- Hash مدیا
- شناسه پیام یا Forward Origin

#### تکرار معنایی

برای تشخیص پست‌هایی که با متن متفاوت، مفهوم مشابهی دارند باید از سرویس هوش مصنوعی یا Embedding استفاده شود.

نتیجه بررسی باید شامل موارد زیر باشد:

- تکراری است یا خیر
- درصد شباهت
- شناسه پست مشابه
- دلیل تشخیص
- روش تشخیص
- مدل استفاده‌شده

آستانه شباهت باید از فایل تنظیمات قابل تغییر باشد.

در صورت تکراری تشخیص داده شدن محتوا، وضعیت پست باید ثبت شود و ادامه Pipeline بر اساس تنظیمات متوقف یا به بررسی دستی منتقل شود.

### 5.10 پاک‌سازی و بازنویسی متن

پیش از ارسال پست برای تایید، متن باید پاک‌سازی شود.

ترتیب عملیات باید به شکل زیر باشد:

1. Username یا شناسه کانال مبدا با Username کانال مقصد جایگزین شود.
1. Username کانال مقصد محافظت شود تا در مرحله حذف پاک نشود.
1. سایر Usernameهای تلگرام حذف شوند.
1. لینک‌های تلگرام نامرتبط حذف شوند.
1. فاصله‌ها و خطوط خالی اضافی اصلاح شوند.
1. Entityهای متن مجدداً محاسبه شوند.
1. Premium Emojiها حفظ شوند.

الگوهای زیر باید شناسایی و حذف شوند:

- @username
- t.me/username
- telegram.me/username
- https://t.me/username
- http://t.me/username
- https://telegram.me/username
- لینک مستقیم پست مانند https://t.me/channel/123
- لینک‌های دارای Query String

حذف لینک یا Username نباید باعث چسبیدن کلمات به یکدیگر یا ایجاد خطوط خالی غیرعادی شود.

اگر پست برای چند کانال مقصد انتخاب شود، متن نهایی باید برای هر مقصد به‌صورت مستقل تولید شود؛ زیرا Username مقصدها ممکن است متفاوت باشد.

متن اصلی نباید در دیتابیس بازنویسی شود. نسخه اصلی و نسخه پردازش‌شده باید جداگانه نگهداری شوند.

### 5.11 دسته‌بندی پست

هر پست باید یک دسته‌بندی داشته باشد.

دسته‌بندی می‌تواند بر اساس یکی از روش‌های زیر تعیین شود:

- دسته‌بندی پیش‌فرض کانال مبدا
- قوانین کلمه کلیدی
- سرویس هوش مصنوعی
- انتخاب دستی مدیر

نمونه دسته‌بندی‌ها:

- اخبار فناوری
- هوش مصنوعی
- امنیت
- اینترنت و فیلترینگ
- پیام‌رسان‌ها
- آموزش
- معرفی ابزار
- سرگرمی
- اقتصادی
- سایر

نام دسته‌بندی باید در هدر پست پیشنهادی نمایش داده شود.

### 5.12 ارسال پست برای مدیران

پس از تکمیل Pipeline، پست باید به کانال، گروه یا گفت‌وگوی تایید محتوا ارسال شود.

پیام پیشنهادی باید شامل دو بخش باشد:

#### هدر متادیتا

هدر باید حداقل اطلاعات زیر را نمایش دهد:

- نام کانال مبدا
- Username کانال مبدا
- شناسه عددی کانال مبدا
- زمان انتشار در کانال مبدا
- زمان دریافت توسط سیستم
- دسته‌بندی
- وضعیت تبلیغاتی بودن
- نتیجه بررسی تکراری بودن
- امتیاز هوش مصنوعی
- شناسه داخلی پست
- وضعیت فعلی پست

نمونه ساختار:

```text
منبع: Example Channel
شناسه: @example_channel
زمان انتشار: ۱۴۰۵/۰۴/۱۹ - ۱۸:۳۵
دسته‌بندی: اخبار فناوری
امتیاز هوش مصنوعی: در انتظار بررسی
وضعیت: آماده تایید
```

#### محتوای پست

متن، Caption و مدیای اصلی باید بعد از هدر ارسال شوند.

هدر نباید وارد متن نهایی کانال مقصد شود.

اطلاعاتی که فقط برای مدیران هستند باید از محتوای قابل انتشار جدا نگهداری شوند.

### 5.13 مدیران مجاز

شناسه عددی مدیران در فایل تنظیمات ثبت می‌شود.

ربات فقط باید به کاربران مجاز پاسخ دهد.

تمام فرمان‌ها و Callbackها باید موارد زیر را بررسی کنند:

- شناسه عددی کاربر
- فعال بودن مدیر
- سطح دسترسی مدیر
- وضعیت فعلی پست
- مجاز بودن عملیات
- مجاز بودن کانال مقصد

پیام کاربران غیرمجاز باید نادیده گرفته شود یا یک پیام دسترسی غیرمجاز دریافت کنند.

اطلاعات حساس مربوط به مدیران نباید در Logها چاپ شود.

### 5.14 دکمه‌های کانال‌های مقصد

برای هر کانال مقصد باید یک سطر دو ستونی از دکمه‌ها نمایش داده شود.

نمونه:

```json
[ زمان‌بندی Kingofilter ] [ فوری Kingofilter ]
[ زمان‌بندی Channel B ]  [ فوری Channel B ]
```

ترتیب دکمه‌ها باید در تمام پیام‌ها ثابت باشد.

دکمه‌ها باید بر اساس کانال‌های مقصد تعریف‌شده در تنظیمات ساخته شوند.

در صورت زیاد بودن کانال‌های مقصد، سیستم باید محدودیت اندازه Callback Data و تعداد دکمه‌های تلگرام را در نظر بگیرد.

Callback Data نباید شامل اطلاعات طولانی یا قابل جعل باشد. بهتر است فقط شامل یک شناسه کوتاه، نوع عملیات و شناسه مقصد باشد.

### 5.15 رفتار Toggle دکمه‌ها

تمام دکمه‌های انتخاب مقصد باید رفتار Toggle داشته باشند.

مثلاً:

- کلیک اول روی «زمان‌بندی Kingofilter» باعث انتخاب آن مقصد برای انتشار زمان‌بندی‌شده شود.
- کلیک دوم روی همان دکمه انتخاب را لغو کند.
- کلیک روی «فوری Kingofilter» حالت انتشار فوری را فعال کند.
- اگر همان مقصد قبلاً روی حالت زمان‌بندی بوده است، انتخاب فوری باید حالت زمان‌بندی را جایگزین کند.
- یک مقصد نباید هم‌زمان در حالت فوری و زمان‌بندی باشد.

حالت‌های هر مقصد می‌توانند شامل موارد زیر باشند:

- انتخاب‌نشده
- انتخاب‌شده برای انتشار فوری
- انتخاب‌شده برای انتشار زمان‌بندی‌شده
- منتشرشده
- زمان‌بندی‌شده
- لغوشده
- خطا در انتشار

ظاهر دکمه باید وضعیت فعلی را نشان دهد. برای مثال:

```text
✅ فوری Kingofilter
🕒 زمان‌بندی Kingofilter
❌ Kingofilter
```

تغییر وضعیت باید ابتدا در دیتابیس ثبت شود و سپس تمام پیام‌های تایید مربوط به مدیران بروزرسانی شوند.

### 5.16 هماهنگی میان مدیران

اگر پست پیشنهادی به چند مدیر ارسال شده باشد، تغییر وضعیت توسط یک مدیر باید برای سایر مدیران نیز نمایش داده شود.

برای این منظور، شناسه پیام تایید ارسال‌شده برای هر مدیر باید ذخیره شود.

پس از هر عملیات:

1. وضعیت جدید در MongoDB ذخیره شود.
1. تمام پیام‌های تایید مرتبط پیدا شوند.
1. متن هدر و دکمه‌های آن‌ها بروزرسانی شوند.
1. دکمه‌های قبلی بدون دلیل حذف نشوند.
1. اگر بروزرسانی یک پیام شکست خورد، سایر پیام‌ها همچنان بروزرسانی شوند.
1. خطا برای Retry در دیتابیس ثبت شود.

برای جلوگیری از Race Condition، عملیات تغییر وضعیت باید Atomic باشد.

اگر دو مدیر هم‌زمان روی یک دکمه کلیک کنند، آخرین وضعیت معتبر دیتابیس باید مبنای نمایش قرار گیرد.

### 5.17 انتشار فوری

وقتی مدیر گزینه انتشار فوری را انتخاب می‌کند:

- وضعیت پست بررسی شود.
- دسترسی مدیر بررسی شود.
- دسترسی حساب Premium به کانال مقصد بررسی شود.
- متن مخصوص کانال مقصد تولید شود.
- مدیا و Entityها آماده شوند.
- پست با حساب Telegram Premium منتشر شود.
- شناسه پیام منتشرشده ذخیره شود.
- زمان انتشار ثبت شود.
- وضعیت دکمه‌ها برای همه مدیران بروزرسانی شود.

در صورت خطا:

- خطا در دیتابیس ثبت شود.
- وضعیت پست به PublishFailed تغییر کند.
- مدیران پیام خطای کوتاه دریافت کنند.
- Retry خودکار فقط به تعداد محدود انجام شود.
- از انتشار تکراری جلوگیری شود.

### 5.18 انتشار زمان‌بندی‌شده

وقتی مدیر گزینه زمان‌بندی را انتخاب می‌کند، زمان انتشار باید بر اساس صف کانال مقصد محاسبه شود.

قانون زمان‌بندی:

- اگر هیچ پست زمان‌بندی‌شده‌ای برای کانال مقصد وجود ندارد، پست برای ۵ دقیقه بعد زمان‌بندی شود.
- اگر پست دیگری در صف وجود دارد، زمان این پست باید ۵ دقیقه بعد از آخرین پست زمان‌بندی‌شده آن کانال باشد.
- صف زمان‌بندی هر کانال مقصد باید مستقل باشد.
- فاصله پنج‌دقیقه‌ای باید از فایل تنظیمات قابل تغییر باشد.

مثال:

```text
آخرین پست زمان‌بندی‌شده Kingofilter: ساعت 18:20
پست جدید: ساعت 18:25

آخرین پست زمان‌بندی‌شده Channel B: وجود ندارد
پست جدید: پنج دقیقه بعد از زمان فعلی
```

زمان‌بندی باید در دیتابیس ذخیره شود و با Restart شدن برنامه از بین نرود.

پس از اجرای مجدد برنامه، تمام Jobهای اجرا‌نشده باید از MongoDB بازیابی شوند.

سیستم باید از اجرای دوباره یک Job جلوگیری کند.

برای هر Job باید موارد زیر ذخیره شود:

- شناسه پست
- کانال مقصد
- زمان برنامه‌ریزی‌شده
- وضعیت Job
- تعداد تلاش
- زمان آخرین تلاش
- زمان اجرای موفق
- شناسه پیام منتشرشده
- خطای آخر

### 5.19 لغو زمان‌بندی

اگر مدیر دوباره روی دکمه زمان‌بندی کلیک کند:

- Job مربوط به آن مقصد لغو شود.
- وضعیت Job در دیتابیس بروزرسانی شود.
- پست نباید در زمان تعیین‌شده ارسال شود.
- دکمه‌های تمام مدیران بروزرسانی شوند.

لغو یک پست نباید لزوماً باعث جابه‌جایی زمان تمام پست‌های بعدی شود.

رفتار فشرده‌سازی مجدد صف باید از تنظیمات قابل کنترل باشد:

- حفظ زمان قبلی سایر پست‌ها
- محاسبه مجدد زمان پست‌های بعدی

حالت پیش‌فرض بهتر است حفظ زمان پست‌های بعدی باشد تا زمان انتشار آن‌ها بدون اطلاع مدیر تغییر نکند.

### 5.20 امتیازدهی هوش مصنوعی

در صورت فعال بودن امتیازدهی، پست نباید بلافاصله امتیاز دریافت کند.

حداقل ۲۰ دقیقه پس از زمان انتشار پست در کانال مبدا، سیستم باید پست را برای ارزیابی به سرویس هوش مصنوعی ارسال کند.

فاصله ۲۰ دقیقه‌ای باید از فایل تنظیمات قابل تغییر باشد.

نتیجه ارزیابی می‌تواند شامل موارد زیر باشد:

- امتیاز کلی از ۰ تا ۱۰۰
- احتمال جذاب بودن محتوا
- احتمال تعامل کاربران
- کیفیت تیتر
- تازگی محتوا
- ارزش خبری
- کیفیت نگارش
- دلیل کوتاه امتیاز

پس از دریافت امتیاز:

1. نتیجه در دیتابیس ذخیره شود.
1. هدر پیام تایید ویرایش شود.
1. امتیاز جدید نمایش داده شود.
1. دکمه‌های Callback حذف نشوند.
1. وضعیت انتخاب کانال‌ها تغییر نکند.
1. تمام نسخه‌های پیام ارسال‌شده برای مدیران بروزرسانی شوند.

اگر مدیر پیش از تکمیل امتیازدهی، پست را منتشر کرده باشد، نتیجه AI همچنان می‌تواند در دیتابیس ذخیره شود؛ اما نیازی به تغییر پیام منتشرشده در کانال مقصد نیست.

### 5.21 Pipeline کامل فاز اول

Pipeline پردازش باید به ترتیب زیر باشد:

```text
دریافت پست‌های روز جاری از کانال مبدا
↓
دریافت لحظه‌ای پست‌های جدید
↓
بررسی یکتا بودن پیام
↓
ذخیره نسخه اولیه در MongoDB
↓
دریافت و ذخیره مدیا
↓
تجمیع پیام‌های Media Group
↓
بررسی تبلیغاتی بودن محتوا
↓
بررسی تکراری بودن دقیق
↓
بررسی تکراری بودن معنایی
↓
تعیین دسته‌بندی
↓
پاک‌سازی Usernameها و لینک‌های تلگرام
↓
بازسازی Telegram Entityها
↓
ارسال پست به مدیران برای تایید
↓
ایجاد دکمه‌های کانال‌های مقصد
↓
انتظار برای تصمیم مدیر
↓
انتشار فوری یا ثبت در صف زمان‌بندی
↓
امتیازدهی هوش مصنوعی پس از زمان تعیین‌شده
↓
ویرایش هدر پیام مدیران بدون حذف دکمه‌ها
```

هر مرحله باید وضعیت مخصوص خود را داشته باشد تا در صورت خطا، پردازش از مرحله مناسب ادامه پیدا کند و کل Pipeline از ابتدا اجرا نشود.

## 6. فاز دوم: مدیریت پست‌های تبلیغاتی زمان‌بندی‌شده

در این فاز، سیستم باید امکان انتشار خودکار پست‌های تبلیغاتی از پیش تعیین‌شده را فراهم کند.

URL پست‌های تبلیغاتی و زمان‌های انتشار آن‌ها در فایل `configuration.json` تعریف می‌شوند.

برای هر تبلیغ باید بتوان موارد زیر را تنظیم کرد:

- نام تبلیغ
- فعال یا غیرفعال بودن
- URL پست مبدا
- کانال مبدا
- کانال‌های مقصد
- روزهای هفته
- چند زمان مختلف در هر شبانه‌روز
- تاریخ شروع
- تاریخ پایان
- منطقه زمانی
- نحوه انتشار
- اولویت
- فاصله از پست قبلی
- رفتار در صورت خطا
- حداکثر دفعات Retry

نمونه مفهومی:

```json
{
  "name": "Daily Store Advertisement",
  "enabled": true,
  "source_post_url": "https://t.me/example/123",
  "destination_channels": [
    "kingofilter",
    "channel_b"
  ],
  "days_of_week": [
    "saturday",
    "monday",
    "wednesday"
  ],
  "times": [
    "10:00",
    "16:30",
    "22:00"
  ],
  "timezone": "Asia/Tehran"
}
```

### 6.1 دریافت پست تبلیغاتی

سیستم باید پست تبلیغاتی را از URL تعیین‌شده دریافت کند.

در صورت آلبوم بودن پست، تمام اعضای آلبوم باید دریافت شوند.

متن، مدیا، Entityها و Premium Emojiها باید حفظ شوند.

اطلاعات پست تبلیغاتی می‌تواند در اولین اجرا Cache شود تا برای هر انتشار دوباره از تلگرام دریافت نشود.

در صورت ویرایش پست مبدا، رفتار بروزرسانی باید قابل تنظیم باشد:

- استفاده از آخرین نسخه پست
- استفاده از نسخه Cacheشده
- بروزرسانی Cache در بازه زمانی مشخص

### 6.2 زمان‌بندی چندگانه

هر تبلیغ می‌تواند در یک روز چند زمان انتشار داشته باشد.

برای مثال:

```text
10:00
15:30
21:00
```

سیستم باید تمام زمان‌ها را به Job مستقل تبدیل کند.

Jobها باید در MongoDB نگهداری شوند تا با Restart شدن برنامه از بین نروند.

زمان‌بندی باید بر اساس منطقه زمانی تنظیم‌شده انجام شود و تغییر ساعت سرور نباید باعث انتشار اشتباه شود.

### 6.3 جلوگیری از انتشار تکراری تبلیغ

برای هر تبلیغ باید ثبت شود که در کدام تاریخ، ساعت و کانال منتشر شده است.

ترکیب زیر می‌تواند یکتا باشد:

- شناسه تبلیغ
- کانال مقصد
- تاریخ اجرای برنامه‌ریزی‌شده
- ساعت اجرای برنامه‌ریزی‌شده

در صورت Restart یا Retry، تبلیغ نباید دوباره برای همان Slot زمانی منتشر شود.

### 6.4 تداخل با صف محتوای عادی

رفتار پست تبلیغاتی در صورت تداخل با پست‌های عادی باید قابل تنظیم باشد.

حالت‌های پیشنهادی:

- تبلیغ در زمان دقیق خود منتشر شود.
- پست عادی به بعد از تبلیغ منتقل شود.
- تبلیغ تا اولین زمان خالی به تعویق بیفتد.
- حداقل فاصله زمانی میان تبلیغ و پست عادی رعایت شود.

برای هر کانال مقصد باید بتوان حداقل فاصله زمانی میان دو انتشار را تعیین کرد.

### 6.5 گزارش اجرای تبلیغات

سیستم باید نتیجه انتشار هر تبلیغ را ثبت کند:

- زمان برنامه‌ریزی‌شده
- زمان واقعی انتشار
- کانال مقصد
- شناسه پیام منتشرشده
- وضعیت موفق یا ناموفق
- تعداد Retry
- خطای آخر
- اختلاف زمان اجرا با زمان برنامه‌ریزی‌شده

مدیران باید بتوانند گزارش مختصری از تبلیغات امروز، تبلیغات آینده و خطاهای اخیر دریافت کنند.

## 7. فاز سوم پیشنهادی: تحلیل و اولویت‌بندی هوشمند محتوا


تاریخچه تغییر وضعیت‌ها نیز بهتر است نگهداری شود.

برای هر تغییر وضعیت موارد زیر ثبت شوند:

- وضعیت قبلی
- وضعیت جدید
- زمان تغییر
- مدیر یا سرویس ایجادکننده تغییر
- دلیل تغییر
- اطلاعات تکمیلی

## 11. پایپ‌لاین سرویس‌های هوش مصنوعی و سیستم Fallback

برنامه از چند سرویس یا مدل هوش مصنوعی رایگان استفاده می‌کند. این سرویس‌ها باید به‌صورت یک پایپ‌لاین اولویت‌بندی‌شده فراخوانی شوند تا در صورت از دسترس خارج شدن، محدودیت سهمیه یا خطای یک سرویس، پردازش برنامه متوقف نشود.

منطق اصلی برنامه نباید مستقیماً به یک ارائه‌دهنده خاص هوش مصنوعی وابسته باشد.

تمام سرویس‌های هوش مصنوعی باید از طریق یک Interface یا Abstraction مشترک در اختیار لایه Application قرار بگیرند.

نمونه عملیات‌هایی که باید از این پایپ‌لاین استفاده کنند:

- تشخیص تبلیغاتی بودن پست
- تشخیص محتوای تکراری معنایی
- دسته‌بندی محتوا
- امتیازدهی پست
- خلاصه‌سازی
- بازنویسی متن
- تشخیص موضوع
- تولید عنوان
- ترجمه محتوا
- تحلیل کیفیت محتوا

### 11.1 ترتیب اولویت ارائه‌دهندگان

فهرست سرویس‌ها و مدل‌های هوش مصنوعی باید در فایل `configuration.json` تعریف شود.

هر سرویس باید دارای یک مقدار اولویت باشد.

سیستم باید ابتدا سرویس دارای بالاترین اولویت را فراخوانی کند.

اگر سرویس اول نتوانست پاسخ معتبر تولید کند، سیستم باید به‌صورت خودکار سرویس بعدی را فراخوانی کند و این روند تا زمان دریافت پاسخ معتبر یا پایان فهرست ادامه پیدا کند.

نمونه ترتیب:

```text
Provider A
↓ در صورت شکست
Provider B
↓ در صورت شکست
Provider C
↓ در صورت شکست
Provider D
```

ترتیب سرویس‌ها باید بدون تغییر کد و فقط از طریق فایل تنظیمات قابل تغییر باشد.

### 11.2 تنظیمات هر Provider

برای هر ارائه‌دهنده هوش مصنوعی باید بتوان موارد زیر را تنظیم کرد:

- نام Provider
- فعال یا غیرفعال بودن
- اولویت
- Base URL
- نام مدل
- API Key
- Timeout
- حداکثر تعداد Retry
- فاصله بین Retryها
- محدودیت تعداد درخواست
- محدودیت هم‌زمانی درخواست‌ها
- حداکثر تعداد Token
- Temperature
- نوع عملیات‌های پشتیبانی‌شده
- قالب درخواست
- قالب پاسخ
- وضعیت استفاده از Streaming
- مدت زمان غیرفعال ماندن پس از خطا
- مدت زمان غیرفعال ماندن پس از Rate Limit

نمونه مفهومی:

```json
{
  "ai": {
    "providers": [
      {
        "name": "provider-a",
        "enabled": true,
        "priority": 1,
        "base_url": "https://api.example-a.com",
        "model": "free-model-a",
        "timeout_seconds": 30,
        "max_retries": 2,
        "cooldown_seconds": 60,
        "supported_tasks": [
          "advertisement_detection",
          "categorization",
          "scoring"
        ]
      },
      {
        "name": "provider-b",
        "enabled": true,
        "priority": 2,
        "base_url": "https://api.example-b.com",
        "model": "free-model-b",
        "timeout_seconds": 45,
        "max_retries": 1,
        "cooldown_seconds": 120,
        "supported_tasks": [
          "advertisement_detection",
          "duplicate_detection",
          "categorization",
          "scoring"
        ]
      }
    ]
  }
}
```

API Keyها نباید مستقیماً در فایل تنظیمات Commitشده قرار بگیرند و باید از Environment Variable یا Secret Manager خوانده شوند.

### 11.3 روند فراخوانی پایپ‌لاین

برای هر درخواست هوش مصنوعی، سیستم باید مراحل زیر را اجرا کند:

1. نوع عملیات مشخص شود.
1. Providerهای فعال که آن عملیات را پشتیبانی می‌کنند انتخاب شوند.
1. Providerها بر اساس اولویت مرتب شوند.
1. سلامت و وضعیت Cooldown هر Provider بررسی شود.
1. اولین Provider معتبر فراخوانی شود.
1. پاسخ دریافت‌شده اعتبارسنجی شود.
1. در صورت معتبر بودن پاسخ، Pipeline متوقف شود.
1. در صورت شکست، دلیل خطا ثبت شود.
1. Provider بعدی فراخوانی شود.
1. این روند تا دریافت پاسخ معتبر یا تمام شدن Providerها ادامه پیدا کند.

نمونه:

```text
ایجاد درخواست دسته‌بندی
↓
انتخاب Providerهای پشتیبان دسته‌بندی
↓
مرتب‌سازی بر اساس اولویت
↓
فراخوانی Provider اول
↓
Timeout یا پاسخ نامعتبر
↓
ثبت خطا
↓
فراخوانی Provider دوم
↓
پاسخ معتبر
↓
اعتبارسنجی و ذخیره نتیجه
```

### 11.4 شرایط شکست Provider

در موارد زیر پاسخ Provider ناموفق در نظر گرفته می‌شود و سیستم باید به Provider بعدی منتقل شود:

- Timeout
- قطع اتصال
- خطای DNS
- خطای SSL
- پاسخ HTTP ناموفق
- خطای 429 Too Many Requests
- پایان سهمیه رایگان
- نامعتبر بودن API Key
- در دسترس نبودن مدل
- پاسخ خالی
- پاسخ ناقص
- JSON نامعتبر
- رعایت نشدن Schema پاسخ
- پاسخ خارج از محدوده مقادیر مورد انتظار
- پاسخ غیرمرتبط
- خطای داخلی Provider
- عدم پشتیبانی Provider از عملیات موردنظر

برای خطاهای احراز هویت مانند API Key نامعتبر، Provider نباید دائماً Retry شود و باید تا اصلاح تنظیمات غیرفعال تلقی شود.

### 11.5 اعتبارسنجی پاسخ

صرف دریافت پاسخ HTTP موفق به معنی موفق بودن عملیات نیست.

پاسخ هر Provider باید پیش از استفاده اعتبارسنجی شود.

اعتبارسنجی باید شامل موارد زیر باشد:

- خالی نبودن پاسخ
- معتبر بودن JSON
- وجود تمام فیلدهای ضروری
- صحیح بودن نوع داده‌ها
- قرار داشتن امتیازها در محدوده معتبر
- معتبر بودن مقادیر Enum
- نبودن متن اضافی خارج از ساختار مورد انتظار
- مرتبط بودن پاسخ با عملیات درخواست‌شده

برای هر عملیات باید یک Schema مستقل تعریف شود.

نمونه پاسخ تشخیص تبلیغات:

```json
{
  "is_advertisement": false,
  "confidence": 0.93,
  "reason": "The post reports a technology update and contains no promotional call to action."
}
```

اگر پاسخ قابل اصلاح باشد، سیستم می‌تواند یک بار تلاش کند آن را Parse یا Repair کند. اگر همچنان نامعتبر بود، باید Provider بعدی فراخوانی شود.

### 11.6 Retry داخلی و Fallback

Retry یک Provider با Fallback میان Providerها متفاوت است.

ترتیب پیشنهادی:

```text
فراخوانی Provider A
↓
Retry محدود Provider A
↓ در صورت شکست کامل
فراخوانی Provider B
↓
Retry محدود Provider B
↓ در صورت شکست کامل
فراخوانی Provider C
```

تعداد Retry هر Provider باید محدود و از تنظیمات قابل تغییر باشد.

برای خطاهایی مانند Timeout یا خطای موقت شبکه می‌توان Retry انجام داد.

برای خطاهایی مانند موارد زیر بهتر است بدون Retry به Provider بعدی منتقل شد:

- API Key نامعتبر
- مدل پیدا نشد
- سهمیه روزانه تمام شده
- عملیات پشتیبانی نمی‌شود
- حساب مسدود شده است

Retryها باید دارای Exponential Backoff باشند.

### 11.7 مدیریت Rate Limit

از آنجا که سرویس‌های رایگان معمولاً محدودیت تعداد درخواست دارند، برنامه باید Rate Limit هر Provider را مدیریت کند.

سیستم باید بتواند موارد زیر را کنترل کند:

- تعداد درخواست در دقیقه
- تعداد درخواست در ساعت
- تعداد درخواست در روز
- تعداد درخواست هم‌زمان
- تعداد Token مصرف‌شده
- زمان ریست شدن سهمیه
- خطاهای 429
- Headerهای مربوط به Rate Limit

اگر یک Provider به محدودیت رسیده باشد:

- درخواست جدید نباید به آن ارسال شود.
- Provider باید موقتاً وارد وضعیت Cooldown شود.
- سیستم باید از Provider بعدی استفاده کند.
- زمان تقریبی آزاد شدن Provider ثبت شود.

استفاده از Provider باید پیش از ارسال درخواست Reserve شود تا چند Worker به‌صورت هم‌زمان از محدودیت تعیین‌شده عبور نکنند.

### 11.8 Circuit Breaker

برای جلوگیری از ارسال مداوم درخواست به یک سرویس خراب، هر Provider باید Circuit Breaker مستقل داشته باشد.

وضعیت‌های Circuit Breaker:

- Closed: Provider سالم است و درخواست دریافت می‌کند.
- Open: Provider موقتاً غیرفعال است.
- HalfOpen: یک درخواست آزمایشی برای بررسی بازیابی Provider ارسال می‌شود.

اگر تعداد خطاهای متوالی یک Provider از حد مشخص عبور کند:

- Circuit آن باز شود.
- Provider برای مدت مشخصی فراخوانی نشود.
- درخواست‌ها به Provider بعدی منتقل شوند.

پس از پایان مدت Cooldown، یک درخواست آزمایشی ارسال شود.

اگر موفق بود، Provider دوباره فعال شود. در غیر این صورت، Circuit همچنان باز باقی بماند.

### 11.9 تنظیم Provider بر اساس نوع عملیات

لازم نیست تمام عملیات‌ها از یک ترتیب Provider یکسان استفاده کنند.

برای هر نوع عملیات باید بتوان ترتیب جداگانه‌ای تعریف کرد.

مثلاً:

```text
تشخیص تبلیغات:
Provider A → Provider B → Provider C

تشخیص محتوای تکراری:
Provider C → Provider B

امتیازدهی:
Provider B → Provider D → Provider A

بازنویسی:
Provider D → Provider A
```

این تنظیم باید کاملاً Config-driven باشد.

یک Provider فقط زمانی باید انتخاب شود که عملیات موردنظر را پشتیبانی کند.

### 11.10 Fallback میان مدل‌های یک Provider

یک Provider ممکن است چند مدل رایگان داشته باشد.

سیستم باید بتواند پیش از رفتن به Provider بعدی، مدل‌های جایگزین همان Provider را نیز امتحان کند.

نمونه:

```text
Provider A / Model 1
↓
Provider A / Model 2
↓
Provider B / Model 1
↓
Provider C / Model 1
```

ترتیب Providerها و مدل‌ها باید از فایل تنظیمات قابل تعریف باشد.

هر ترکیب Provider و Model باید وضعیت سلامت، محدودیت و آمار مستقل داشته باشد.

### 11.11 یکسان‌سازی خروجی Providerها

هر سرویس ممکن است پاسخ متفاوتی برگرداند.

برای جلوگیری از وابستگی منطق برنامه به ساختار هر Provider، پاسخ تمام سرویس‌ها باید به یک مدل داخلی استاندارد تبدیل شود.

نمونه مدل نتیجه:

```text
AIResult
- success
- task_type
- provider_name
- model_name
- result
- confidence
- reason
- prompt_version
- latency
- input_tokens
- output_tokens
- attempt_number
- fallback_count
- created_at

```

لایه Application فقط باید مدل استاندارد داخلی را دریافت کند و نباید از ساختار اختصاصی Providerها اطلاع داشته باشد.

### 11.12 رفتار در صورت شکست تمام Providerها

اگر تمام Providerهای فعال ناموفق باشند، سیستم نباید نتیجه جعلی تولید کند.

در این حالت باید:

- وضعیت عملیات به AIProcessingFailed تغییر کند.
- خطای هر Provider ثبت شود.
- تعداد Providerهای امتحان‌شده ثبت شود.
- درخواست برای Retry آینده ذخیره شود.
- در صورت نیاز، پست برای بررسی دستی مدیر ارسال شود.
- سایر بخش‌های غیرضروری Pipeline متوقف نشوند.

رفتار نهایی باید برای هر عملیات قابل تنظیم باشد.

مثلاً برای تشخیص تبلیغات:

```text
در صورت شکست همه Providerها:
- ارسال برای بررسی دستی

یا
- ادامه پردازش

یا
- توقف پردازش

برای دسته‌بندی:
در صورت شکست همه Providerها:
- استفاده از دسته‌بندی پیش‌فرض کانال مبدا

برای امتیازدهی:
در صورت شکست همه Providerها:
- نمایش «امتیاز در دسترس نیست»
- Retry در زمان بعد

```

### 11.13 صف پردازش هوش مصنوعی

درخواست‌های AI بهتر است مستقیماً در Handler دریافت پیام اجرا نشوند.

برای جلوگیری از کند شدن خزش، هر عملیات AI باید به‌صورت Job در صف ثبت شود.

هر Job باید شامل موارد زیر باشد:

- شناسه Job
- شناسه پست
- نوع عملیات
- اولویت Job
- Providerهای مجاز
- وضعیت
- تعداد تلاش کلی
- Provider فعلی
- مدل فعلی
- زمان اجرای بعدی
- زمان ایجاد
- زمان آخرین تلاش
- نتیجه نهایی
- خطای آخر

وضعیت‌های پیشنهادی Job:

```text
Pending
Processing
WaitingForRetry
Completed
AllProvidersFailed
Cancelled
Expired
```

Jobها باید در MongoDB ذخیره شوند تا با Restart شدن برنامه از بین نروند.

### 11.14 اولویت‌بندی Jobهای هوش مصنوعی

درخواست‌های AI باید بتوانند اولویت متفاوت داشته باشند.

نمونه اولویت:

```text
تشخیص تبلیغات: بالا
تشخیص تکراری بودن: بالا
دسته‌بندی: متوسط
امتیازدهی: متوسط
خلاصه‌سازی: پایین
تحلیل‌های آماری: پایین
```

Jobهای مربوط به انتشار محتوا باید نسبت به تحلیل‌های غیرضروری در اولویت بالاتری قرار بگیرند.

### 11.15 Cache نتایج هوش مصنوعی

برای کاهش مصرف سهمیه رایگان، نتایج عملیات AI باید قابل Cache باشند.

کلید Cache می‌تواند از موارد زیر ساخته شود:

- Hash متن Normalizeشده
- نوع عملیات
- نسخه Prompt
- نسخه Schema
- زبان محتوا

اگر همان محتوا با همان Prompt و همان عملیات قبلاً پردازش شده باشد، سیستم باید در صورت معتبر بودن Cache از نتیجه قبلی استفاده کند.

مدت اعتبار Cache باید برای هر عملیات قابل تنظیم باشد.

تغییر نسخه Prompt یا Schema باید Cache قبلی را نامعتبر کند.

### 11.16 ثبت Prompt Version

تمام Promptهای مورد استفاده باید نسخه‌بندی شوند.

برای هر نتیجه AI باید موارد زیر ثبت شود:

- نسخه Prompt
- متن یا Hash Prompt
- Provider
- Model
- پارامترهای درخواست
- زمان اجرا
- پاسخ خام
- پاسخ Parseشده

ذخیره پاسخ خام باید قابل تنظیم باشد، زیرا ممکن است حجم دیتابیس را افزایش دهد یا حاوی اطلاعات حساس باشد.

### 11.17 لاگ و آمار Providerها

برای هر فراخوانی باید اطلاعات زیر ثبت شود:

- نوع عملیات
- Provider
- مدل
- زمان شروع
- مدت پاسخ
- موفق یا ناموفق بودن
- کد HTTP
- نوع خطا
- تعداد Retry
- تعداد Fallback
- مصرف Token
- استفاده از Cache
- شناسه پست
- شناسه Job

سیستم باید آمار تجمعی هر Provider را نگهداری کند:

- تعداد درخواست
- نرخ موفقیت
- نرخ خطا
- متوسط زمان پاسخ
- تعداد Timeout
- تعداد Rate Limit
- تعداد پاسخ نامعتبر
- تعداد دفعات Fallback
- آخرین زمان موفقیت
- آخرین زمان خطا

این اطلاعات در آینده برای تغییر خودکار ترتیب Providerها قابل استفاده خواهند بود.

### 11.18 عدم پردازش هم‌زمان تکراری

یک پست نباید برای یک عملیات AI به‌صورت هم‌زمان توسط چند Worker پردازش شود.

برای جلوگیری از این مشکل باید از Lock توزیع‌شده، Lease یا عملیات Atomic در MongoDB استفاده شود.

هر Job هنگام دریافت توسط Worker باید برای مدت مشخص Lock شود.

اگر Worker از کار افتاد، Lock باید پس از پایان زمان Lease آزاد شود تا Worker دیگری بتواند Job را ادامه دهد.

### 11.19 معیارهای پذیرش پایپ‌لاین AI

این بخش زمانی کامل محسوب می‌شود که:

1. چند Provider از طریق فایل تنظیمات قابل تعریف باشند.
1. ترتیب Providerها بدون تغییر کد قابل تغییر باشد.
1. Providerهای غیرفعال فراخوانی نشوند.
1. فقط Providerهای پشتیبان یک عملیات انتخاب شوند.
1. در صورت خطای Provider اول، Provider بعدی فراخوانی شود.
1. پاسخ نامعتبر نیز باعث Fallback شود.
1. Retry داخلی هر Provider محدود باشد.
1. خطای 429 باعث Cooldown Provider شود.

## 13. مدیریت خطا و Retry

تمام درخواست‌های خارجی باید Timeout داشته باشند.

Retry باید محدود و دارای فاصله افزایشی باشد.

نمونه سرویس‌هایی که نیاز به Retry دارند:

- Telegram
- MongoDB
- سرویس هوش مصنوعی
- دانلود فایل
- آپلود فایل
- ویرایش پیام مدیران

خطاهای دائمی و موقت باید از یکدیگر تفکیک شوند.

مثلاً:

- Flood Wait تلگرام باید بر اساس زمان اعلام‌شده مدیریت شود.
- دسترسی نداشتن به کانال خطای دائمی محسوب شود.
- قطع موقت اینترنت خطای قابل Retry باشد.
- نامعتبر بودن URL پست تبلیغاتی خطای تنظیمات باشد.

پس از رسیدن Retry به حداکثر مقدار، عملیات باید به Dead Letter Queue یا وضعیت نیازمند بررسی دستی منتقل شود.
